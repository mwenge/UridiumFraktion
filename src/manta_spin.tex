In its simplest terms the vertical order of the black lines detailing the cockpit have been reversed
and this alone is enough to suggest the difference in attitude along the Y co-ordinate.

Examining this detail may also give you a sense of how the sprites themselves are defined. In the diagram
above, for example, it may now be obvious that the black colour is defined by a bit pair of \icode{10} and
that the white colour is defined by a bit pair of \icode{11}. In the diagrams opposite, we can further see
that yellow is defined by the bit paid \icode{01} and the transparent background of the sprite by \icode{00}.

In the diagrams opposite and elsewhere we have given the three bytes that together make up the 12 bit pairs constituting
each of the 21 lines in the sprite, a total of 63 bytes in total. This is the source data used by \textit{Uridium's} 6502
assembler code to build each sprite using the straightforward process of mapping each bit pair to a corresponding
pair of coloured pixels. An important feature of this sprite data is that it is stored contiguously for all sprites in the computer's
RAM, or memory area, starting at address \icode{\$5000}. This is important because the order in which the sprite
data is stored corresponds exactly to the order of each frame in the manta spin animation. 

We can get a better sense of this by observing the RAM address of each sprite in the section of animation sequence
between the ship faced towards and faced away. The first sprite in the sequence is at address \icode{\$5040}, the second
at \icode{\$5080}, and so on. At each step, to get the next frame in the animation all that has to be done is to 
increment the position in RAM we are adressing by \icode{\$40}.

This approach does not hold for the second half of the animation sequence, or at least
not all of it. As we can see below, the final two frames of the sequence reset back
to address \icode{\$5000} to stitch in the start of the animation and complete the
loop.
\begin{figure}[H]
    \centering
    \begin{adjustbox}{width=13.5cm,center}
      \includegraphics[width=15cm]{src/manta_spin/manta_spin_second_half.png}%
    \end{adjustbox}

But this is not a wrinkle the animation needs to worry about. All it needs to do
is loop across the sprites that constitute the full sequence. It is indifferent to the
fact that the first frame in RAM is not a manta sitting flush to our point of view.

Let's take a look at how it is implemented in the routine responsible
for animating the spinning manta. It's thankfully very compact and easy to digest.
We would expect it to iterate over the sprites from \icode{MANTA\_0} to \icode{MANTA\_15}, which is 
exactly what it does - as we noted, all that matters to it is that the sprites are ordered
contiguously in RAM. Selecting the next sprite for preparation (\icode{GetCurrentSprite})
and display (\icode{DisplaySprite}) is a simple matter of incrementing the index
\icode{currentSpriteValue} that points to the current sprite until we reach \icode{MANTA\_16}, 
at which point we reset our \icode{currentSpriteValue} back to \icode{MANTA\_0}.

\begin{lstlisting}
SpinningShipLoop
    JSR CheckInput            ; Increment frameCount and check input.
    LDA firePressed           ; Was 'Fire' pressed?
    BEQ AnimationOver         ; If so, stop animating ship.
    LDA frameCount            ; Is the sequence over?
    BEQ AnimationOver         ; If so, stop animating ship.
    JSR MaybeShowPauseScreen  ; Check if 'Pause' pressed.
    JSR GetCurrentSprite      ; Prepare current sprite.
    INC currentSpriteValue    ; Increment the sprite index.
    LDA currentSpriteValue    ; Store it in 'A' register.
    CMP #MANTA_16             ; Is it MANTA_16 yet?
    BCC Repeat                ; If not, display sprite and loop. 
    LDA #MANTA_00             ; Otherwise, fetch MANTA_00 ..
    STA currentSpriteValue    ; .. and store it as current sprite.
Repeat
    JSR DisplayCurrentSprite  ; Display our selected sprite.
    JMP SpinningShipLoop      ; Loop back to top again.
\end{lstlisting}

But there is one complication we haven't yet accounted for: how does the animation ensure
that when \icode{frameCount} has reached zero, and the animation is over, that we finish with
the Manta facing upwards:
\raisebox{-0.2\totalheight}{\includegraphics[height=0.5cm]{src/manta_spin/MANTA1.png}}?
The answer lies in the initial value that we set for \icode{frameCount}. This is \icode{\$BF} or 191
in decimal: 
\begin{lstlisting}
    LDA #$BF         ; Store 191 in the A register.
    STA frameCount   ; Store it as our frameCount.
\end{lstlisting}
A starting value of 191 for \icode{frameCount} gives us 65 keyframes of animation in total until
we reach 256 and roll around, like a clock, back to zero. With 16 frames for each complete spin of the Manta
this allows us 4 complete spins (64 frames) with one left over before \icode{frameCount} will roll over to
zero. It is this left over frame that will push us past \icode{MANTA\_00} -  
\raisebox{-0.2\totalheight}{\includegraphics[height=0.5cm]{src/manta_spin/MANTA.png}}
- and finish at the desired spot of \icode{MANTA\_1} - 
\raisebox{-0.2\totalheight}{\includegraphics[height=0.5cm]{src/manta_spin/MANTA1.png}}.
